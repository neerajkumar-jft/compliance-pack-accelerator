§2 — Why We Made These Changes

MOTIVATION Hackathon-winner pattern

This team came 4th in a Databricks hackathon; the winners had persona-specific dashboards + scoped agents per role. Replicating that pattern — cleanly, with real governance — is the immediate driver.

MOTIVATION UC-enforced boundaries, not just UI access control

An SA's first question: "Is this actual governance, or just hidden menus?" The answer has to be UC at query time, not workspace ACLs alone. Every persona query is validated by UC against their SELECT grants before execution — independent of how they arrived at the query (dashboard tile, Genie, SQL editor, notebook, API).

MOTIVATION Teammate-reproducible deploy

The POC was originally set up on a single workspace (the original deployer's trial); other teammates reviewing the work had no path to reproduce it. The orchestrator + auto-detected config + the persona_deploy.md guide + the README update close that loop. Anyone on the team can now clone the repo, run one command, and stand up the persona layer on their own workspace.

MOTIVATION Bundle hygiene for CI/CD posture

Databricks CLI v0.288+ is stricter on schema validation. A bundle that fails validate can't be deployed in a CI pipeline, and red warnings during an SA review are a distraction. Cleaning the bundle moves the story from "well yeah, there are some warnings" to "validate is green, ready to ship."

DEFERRED Custom Databricks App (FastAPI portal)

We initially built a custom FastAPI Databricks App with a persona-dropdown landing page and an embedded chat that called databricks-gpt-oss-120b directly. It worked, but Databricks already offers Genie spaces as a native primitive with a stronger governance story (per-space ACLs, native query generation, UC lineage). The custom app was deleted in favor of Genie, with a link-tile on each dashboard for the persona's Genie space. This is simpler, more Databricks-idiomatic, and covers ~80% of the use case at ~5% of the code.
