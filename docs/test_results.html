T2.6 Delta CDF enables live withdrawal propagation

The consent log is published with delta.enableChangeDataFeed = true, so downstream systems can subscribe to every grant, withdrawal, and supersession as it happens — no polling, no full-table diffs. This test demonstrates the CDF capability and traces a withdrawal from the CDF stream all the way to its downstream effect on gold.marketing_eligible_principals.

Step 1 — confirm CDF is enabled

SHOW TBLPROPERTIES compliance_pack.compliance.consent_events_log;

-- Expected: the row `delta.enableChangeDataFeed = true` is present.
-- Also visible: delta.logRetentionDuration = interval 730 days
-- (two-year audit retention for DPDP record-of-processing evidence).

Step 2 — enumerate change operations

SELECT _change_type, COUNT(*) AS n
FROM table_changes('compliance_pack.compliance.consent_events_log', 0)
GROUP BY _change_type
ORDER BY n DESC;

-- Expected on this workspace:
--   insert           1000    (initial load — every consent event)
--   update_preimage  1       (the BEFORE state of the one supersession UPDATE)
--   update_postimage 1       (the AFTER state — superseded_by_event_id set)
-- A downstream subscriber would see every new event as _change_type='insert'
-- in real time, plus any supersession UPDATE paired with pre/post images.

Step 3 — isolate withdrawal events in the CDF stream

SELECT
  _commit_version,
  _commit_timestamp,
  event_id,
  data_principal_id,
  purpose,
  event_type,
  purpose_grant_status
FROM table_changes('compliance_pack.compliance.consent_events_log', 0)
WHERE _change_type = 'insert'
  AND event_type = 'withdrawn'
ORDER BY _commit_timestamp;

-- Expected: ~10 withdrawal events, each with its commit timestamp.
-- customer_04217's marketing_email withdrawal (evt_000003) appears here.
-- This is the feed a downstream cache / marketing platform would consume
-- to invalidate a principal the moment consent is withdrawn.

Step 4 — trace one withdrawal end to end

-- (a) customer_04217's consent journey in order:
SELECT event_type, purpose, purpose_grant_status, event_timestamp, superseded_by_event_id
FROM compliance_pack.compliance.consent_events_log
WHERE data_principal_id = 'customer_04217'
ORDER BY event_timestamp;
-- Expected: 4 rows — the February grants, the declined third_party_sharing,
-- and the April marketing_email withdrawal. The February grant row has
-- superseded_by_event_id = evt_000003 (the supersession UPDATE).

-- (b) Is customer_04217 in marketing_eligible_principals?
SELECT * FROM compliance_pack.gold.marketing_eligible_principals
WHERE data_principal_id = 'customer_04217';
-- Expected: 0 rows. The view filters to latest event per
-- (principal, purpose) with purpose_grant_status='granted'. customer_04217's
-- latest marketing_email event is a withdrawal, so the view excludes them —
-- automatic, no manual cache invalidation needed.
Pass criteria: CDF enabled (Step 1); table_changes() returns insert + update streams (Step 2); withdrawals visible in CDF with commit timestamps (Step 3); customer_04217 is in the consent log with a granted→withdrawn trajectory, AND is NOT in marketing_eligible_principals (Step 4). The last assertion is the propagation — the gold view re-computes on every read, so downstream marketing audiences exclude the principal the moment their withdrawal lands.
Why this matters for DPDP: §8(7) requires that withdrawal take effect without undue delay. Polling-based integrations ("re-sync the marketing list every 4 hours") fail that bar. CDF + latest-state views give sub-minute effective propagation with an auditable change stream — exactly the enforcement posture DPDP envisions.

T2.7 10-language notice generator (pack-driven)

The active regulation pack declares every language the POC covers in languages.yaml. scripts/generate_multilang_notices.py reads the seeded primary-locale notice, calls the foundation model endpoint to translate into each non-seeded language, and MERGEs results into compliance.notice_versions.        CASE WHEN notice_text LIKE '[MACHINE-TRANSLATED%'
            THEN 'machine' ELSE 'human' END AS origin
FROM compliance_pack.compliance.notice_versions

--           7 machine rows (bn-IN, te-IN, mr-IN, gu-IN, kn-IN, ml-IN, pa-IN)

T4.4 Risk and compliance scores

WITH metrics AS (
  SELECT COUNT(DISTINCT source_table) AS total_tables,
         SUM(CASE WHEN sensitivity_tier = 'critical' THEN 1 ELSE 0 END) AS critical_pii,
         SUM(CASE WHEN sensitivity_tier = 'high' THEN 1 ELSE 0 END) AS high_pii
  FROM compliance_pack.compliance.personal_data_register
), gaps AS (
  SELECT COUNT(CASE WHEN severity = 'critical' THEN 1 END) AS critical_gaps
  FROM compliance_pack.silver.compliance_gaps
)
SELECT
  LEAST(100, CAST((m.critical_pii * 25 + m.high_pii * 15 + g.critical_gaps * 20)
    / GREATEST(1, m.total_tables * 2) AS INT)) AS risk_score,
  GREATEST(0, 100 - LEAST(100, CAST((m.critical_pii * 25 + m.high_pii * 15 + g.critical_gaps * 20)
    / GREATEST(1, m.total_tables * 2) AS INT))) AS compliance_score
FROM metrics m CROSS JOIN gaps g;

-- Expected: risk_score > 50 (HIGH), compliance_score < 50

T7.2 Delta time travel works

Demonstrates audit-time reproducibility — answering a regulator's "what did our X look like on date Y" question by querying a previous version of a Delta table.

-- 1. Inspect the version history of a non-masked Delta table.
--    bronze.data_sources is updated each time phase1_bootstrap runs
--    (idempotent MERGE on source_id), so its version count grows
--    every deploy. Each row in DESCRIBE HISTORY is one version.
DESCRIBE HISTORY compliance_pack.bronze.data_sources;

-- 2. Query the latest version vs. the most recent prior version
--    (substitute the actual version numbers from step 1; recent
--    versions only — Delta retains deleted files for 168 hours by
--    default, so very old versions may be unreachable).
SELECT 8 AS v, COUNT(*) AS rows, MAX(updated_at) AS last_update
FROM compliance_pack.bronze.data_sources VERSION AS OF 8
UNION ALL SELECT 7, COUNT(*), MAX(updated_at)
FROM compliance_pack.bronze.data_sources VERSION AS OF 7;

-- Expected: row count consistent (10 sources are stable), last_update
-- timestamp differs between versions (each MERGE writes new updated_at).

-- 3. TIMESTAMP AS OF works the same way — useful when you have an audit
--    request phrased as a date rather than a version number. The timestamp
--    MUST be at or after the table's first commit (column "timestamp" in
--    DESCRIBE HISTORY above). A bare date like '2026-04-28' resolves to
--    midnight, which precedes any same-day deploy and raises
--    DELTA_TIMESTAMP_EARLIER_THAN_COMMIT_RETENTION. Pick a real timestamp
--    from step 1, or use a small relative offset that's guaranteed to land
--    after the latest commit:
SELECT COUNT(*) FROM compliance_pack.bronze.data_sources
  TIMESTAMP AS OF (current_timestamp() - INTERVAL 10 SECONDS);
Important — Unity Catalog limitation: tables that carry a column mask or row-level security policy do not support time travel. Querying silver.customers_tagged VERSION AS OF 0 raises COLUMN_MASKS_FEATURE_NOT_SUPPORTED.TIME_TRAVEL; querying compliance.consent_events_log VERSION AS OF 0 raises ROW_LEVEL_SECURITY_FEATURE_NOT_SUPPORTED.TIME_TRAVEL. This is by design — UC cannot retroactively apply current policies to historical data. For audit reproducibility on masked silver tables, use the audit log (system.access.audit) rather than time travel; for unmasked tables (bronze.data_sources, bronze.compliance_rules, etc.) time travel works as demonstrated above.
Why this matters: Time travel and the access audit log together are how the platform reproduces historical state for regulator inspection. "What did our ingestion-source registry look like yesterday?" is answered by VERSION AS OF or TIMESTAMP AS OF; "who queried which PII column on what date?" is answered from system.access.audit (see Test 10).

  expectation.dataset                                 AS silver_table,
  expectation.name                                    AS expectation,
  expectation.failed_records                          AS failed,
  expectation.passed_records                          AS passed,
  ROUND(
    100.0 * expectation.passed_records /
    NULLIF(expectation.passed_records + expectation.failed_records, 0),
    2

  )
